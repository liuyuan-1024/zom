## 2. 核心设计哲学

1. 语义先于实现：先定义领域语义，再写 API 与渲染。
2. 单一变化原因：每个 crate 只响应一种主要变化。
3. 依赖单向流动：低层不依赖高层，禁止循环依赖。
4. 能力归位：算法在内核层，编排在 runtime，渲染在 gpui。
5. 单一真相源：共享语义类型只在 `zom-protocol` 定义，禁止 1:1 镜像复制。
6. 纯函数优先：复杂逻辑先抽纯函数，再接状态与 UI。
7. 默认可测试：核心行为在无 UI 环境可验证。
8. 默认可观测：关键路径可打点、可对比、可回归。

## 3. Crate 宪章（单一职责声明）

| Crate | 主要职责（Owner） | 禁止拥有 | 主变化原因 |
|---|---|---|---|
| `zom-protocol` | 跨层语义类型、命令协议、稳定契约 | UI 类型、业务状态机、平台实现 | 共享语义演进 |
| `zom-input` | 默认键位策略、快捷键注册、输入解析规则 | 业务状态机、UI 渲染实现 | 输入策略演进 |
| `zom-text` | 文本存储、索引、位置映射 | 命令分发、UI 渲染 | 文本模型与性能 |
| `zom-editor` | 编辑行为、选区/光标、事务、undo/redo | GPUI 类型、窗口布局编排 | 编辑行为语义 |
| `zom-workspace` | 工作台模型（pane/tab/project） | 渲染细节、平台能力 | 工作台模型演进 |
| `zom-runtime` | 命令分发、用例编排、projection | 具体绘制实现 | 跨模块流程协调 |
| `zom-gpui` | 事件适配、组件渲染、UI 绑定 | 文本底层算法、编辑内核 | UI 交互与表现 |
| `apps/zom-desktop` | 启动、注入、运行时配置 | 核心业务规则、编辑算法 | 平台装配与启动 |

## 4. 模块结构哲学（crate 内）

### 4.1 组织原则

1. 按“领域能力”组织，而非按“技术类别”组织。
2. 每个能力必须有单一入口（facade），避免调用方耦合内部细节。
3. UI 层组件不直接承载复杂业务推导，优先抽到纯函数/投影层。

### 4.2 推荐目录模板

```text
src/
  lib.rs                # 唯一对外入口（re-export）
  features/             # 按能力分目录
  projection/           # 可选：展示投影（通常在 runtime）
  api/                  # 可选：稳定门面
  tests/                # 可选：行为与边界测试
README.md               # crate 级职责说明
ARCHITECTURE.md         # crate 级边界与演进约束
```

### 4.3 UI crate（`zom-gpui`）特殊规则

1. 组件文件优先保持“三段式”：`state sync`、`derive`、`render`。
2. 复杂计算必须抽函数（如窗口切片、缓存 key 计算、可视范围推导）。
3. 禁止把编辑内核逻辑临时放在视图文件里“先跑通”。

## 5. 跨 crate 协作哲学

### 5.1 单一路径

统一采用：`keyboard/protocol -> runtime -> projection -> gpui`

1. 用户意图在 `protocol` 表达。
2. 执行编排在 `runtime` 完成。
3. 展示数据通过 `runtime::projection` 投影。
4. `gpui` 只消费投影，不重建语义。

### 5.2 变更落点判定

1. 改的是语义定义：`zom-protocol`
2. 改的是文本结构/映射：`zom-text`
3. 改的是编辑行为：`zom-editor`
4. 改的是跨模块流程：`zom-runtime`
5. 改的是交互与视觉：`zom-gpui`

## 6. 治理执行（轻治理）

### 6.1 文档分工（SoT）

1. 依赖方向、allowed/forbidden、边界硬规则：以 [Crate边界契约](./Crate边界契约.md) 为准。
2. 命名、注释、测试、编码规范：以 [开发规范手册](../standards/开发规范手册.md) 为准。
3. 本文只保留治理触发条件与节奏，不重复抄写通用规则正文。

### 6.2 治理触发条件（本文件）

1. 开发前必须声明“主 crate（唯一）+ 次要 crate（尽量 <= 2）”。
2. 若连续改动 3 个以上 crate，必须先复盘边界，再继续叠加功能。
3. 触及 `zom-protocol` 的 PR 必须附变更理由与影响面。
4. 发现 `zom-gpui` 承载业务推导时，优先回收至 `zom-runtime` / `zom-editor`。
5. 涉及 SoT 规则变化时，必须同步更新对应文档与 `docs/README.md` 映射。

---

## 7. PR 评审清单（建议固化到模板）

1. 是否明确主 crate，且符合其单一变化原因。
2. 是否触及 SoT 规则；若触及，是否同步更新对应文档。
3. 是否通过边界守卫脚本与相关测试。
4. 是否出现上层重复稳定语义、或 UI 层混入内核逻辑。
5. 是否在 PR 中写明风险与回滚路径。

---

## 8. 渐进落地计划

### 阶段 A：建立共识（1 周）

1. 为每个 `zom-xxx` 增加 `ARCHITECTURE.md`（职责、禁止项、依赖说明）。
2. 在 PR 模板中加入“主 crate + 边界检查项”。

### 阶段 B：回收漂移（1~2 周）

1. 优先回收 `zom-gpui` 中的业务推导逻辑到 `runtime/editor`。
2. 合并重复语义定义，统一真相源到 `zom-protocol`。

### 阶段 C：持续治理（持续）

1. 每次跨 3+ crate 的需求必须做边界复盘。
2. 每月做一次“跨 crate 改动分布”回顾，识别新的结构热点。

---

## 9. 成功指标（建议季度跟踪）

1. PR 平均触及 crate 数下降。
2. `zom-gpui` 中业务逻辑占比下降。
3. 边界相关 review 返工次数下降。
4. 新功能从需求到合入的跨模块沟通成本下降。
