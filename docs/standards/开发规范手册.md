## 1. 设计原则

1. 每个需求先明确唯一主 crate；若实现持续跨 3+ crate，先暂停并做边界复盘，再继续开发。
2. 命令语义与共享类型统一收敛到 `zom-protocol`，默认快捷键与输入解析策略统一收敛到 `zom-input`；禁止在上层 crate 做 1:1 镜像语义与冗余转换。
3. UI 层只做事件适配与渲染绑定，不承载编辑内核算法；展示文案统一消费 runtime projection。
4. 架构哲学、职责归属与依赖边界判定不在本文重复定义，统一以 SoT 文档为准（见第 2 节）。

### 1.1 产品与输入原则

1. Zom 的输入设计不采用 Vim 式模式切换作为核心交互模型。
2. 默认输入体验应接近 Zed：直接编辑、低心智负担、尽量减少“当前处于什么模式”的负担。
3. 高频操作必须支持全键盘直达，不要求用户频繁切换到鼠标。
4. 低频操作统一通过命令面板承载，避免为低频功能堆积复杂快捷键体系。
5. 工具栏、提示文案、命名和交互设计中，不应引入 `NORMAL`、`INSERT` 这类 Vim 模式语言，除非未来明确实现可选的 Vim 兼容层。
6. 命令系统表达的是“用户意图”，不是“模式状态”；输入系统表达的是“按键如何映射到意图”，不是“当前进入了哪种输入模式”。

### 1.2 核心术语约定

为了在代码命名、架构设计和日常沟通中保持一致，避免概念混淆，特约定以下核心界面术语（代码中的变量名和结构体名应严格遵守此对应关系）：

* **Workspace（工作台/工作区）**：最外层的运行环境。负责管理所有的窗口、面板和分割布局，是整个 IDE 的顶级容器。
* **Pane（窗格）**：带有标签页和具体内容展示区的独立块。一个 Workspace 可以被分割成多个 Pane（例如常见的左右分屏或上下分屏）。
* **Tab（标签页）**：Pane 顶部的导航条元素，用于在 Pane 内部的多个打开项之间进行切换。

### 1.3 单一命令语义 + 多视图投影

1. `zom-protocol` 采用双层命令模型：`CommandKind`（稳定语义族）+ `CommandInvocation`（运行时调用）。
2. `zom-protocol::command::kind` 是命令元信息单一来源：统一维护 `CommandKind`、`CommandKindSpec`、`CommandMeta`（纯语义目录）。
3. `zom-input` 负责把 `Keystroke` 投影到 `CommandInvocation`（键位视图），并维护默认键位与解析策略。
4. `zom-runtime` 负责 `CommandInvocation` 分发到领域处理与副作用编排（执行视图），并负责把命令元信息投影为 UI 可消费展示数据（展示视图）。
5. `zom-gpui` 的 tooltip、菜单、按钮提示必须只消费 `zom-runtime::projection`，禁止直接读取 `zom-protocol` catalog 的展示投影函数。
6. 禁止在任意上层 crate 重新创建与 `CommandInvocation` 1:1 镜像的语义枚举。
7. 工作台显隐语义统一为：`FocusPanel(target)=显示+聚焦`，`CloseFocused=关闭当前聚焦组件`；禁止再引入 `TogglePanel` 用户语义入口。

### 1.4 命令语义命名规范

1. 命令命名必须满足“动词 + 语义对象”的可预测结构，避免同义词并存。
2. 对“可聚焦且可见性托管”的容器，统一使用 `FocusXxx(target)` 命名，语义固定为“显示并聚焦”。
3. 当前焦点关闭语义统一使用 `CloseFocused`，禁止并行引入 `ClosePanel`、`CloseOverlay`、`HideXxx` 等重复入口。
4. `OpenXxx` 仅用于“入口型动作”或“非焦点容器动作”（如系统选择器、外部对话框）。
5. `StartXxx` 仅用于“过程启动类动作”（如调试、任务、会话），不用于 UI 显隐语义。
6. 导航与选择类动作统一使用 `Select*`、`Activate*`、`Expand*`、`Collapse*` 前缀，不使用等价但不统一的别名。
7. 容器语义命名必须与术语体系一致：`Panel` 表示侧边面板，`Pane` 表示主内容窗格，`Overlay` 表示悬浮层。
8. 若设置页采用悬浮交互，语义优先定义为 `FocusOverlay(OverlayTarget::Settings)`，不再新增等价 `OpenSettings` 主入口。
9. 不允许长期并存等价语义（例如 `OpenSettings` 与 `FocusOverlay(Settings)`）；若迁移期需要 alias，必须标注过渡状态并在后续版本删除。
10. `CommandKindId` 命名采用统一模板：`<domain>.<verb>_<object>[.<target>]`，示例：`workspace.focus_panel.file_tree`、`workspace.focus_overlay.settings`、`workspace.file_tree.select_next`。

## 3. 代码组织规范

### 3.1 模块组织

- 优先按“领域功能”组织，而非按“工具类别”组织。
- 示例（`zom-editor`）：
  - `features/cursor/*`
  - `features/selection/*`
  - `features/history/*`
  - `features/search/*`

### 3.2 API 设计

1. crate 对外暴露统一入口（facade），避免调用方了解内部结构。
2. 参数优先传领域类型，不传裸 `u64/String`（使用强类型 ID）。
3. 明确同步/异步边界，避免隐式阻塞 UI。

### 3.3 错误处理

1. 公共层返回结构化错误，避免字符串拼接错误。
2. 禁止在库代码中滥用 `unwrap/expect`。
3. 禁止使用 `#[allow(...)]`、`#![allow(...)]` 或其他形式主动屏蔽 warning。
4. 出现 warning 时，应修复根因，而不是通过 `allow` 压过去。

### 3.4 注释规范

本规范强制采用 Rust 生态标准的 `rustdoc` 格式，以支持使用 `cargo doc` 自动生成文档，并强调“重意图、轻语法”的注释原则。

1. 全面采用 rustdoc 格式：所有生产代码的文档级注释必须使用 Rust 标准的文档注释格式。
   - 文件/模块级注释：每个非空模块文件（`.rs`）顶部必须使用 `//!` 简述该文件的核心职责或主要功能。
   - 结构/方法级注释：所有公开（`pub` 或 `pub(crate)`）的结构体、枚举、特征、函数，以及核心字段，必须使用 `///` 注释。
2. 普通逻辑说明：函数内部对代码块的解释使用 `//` 即可，不需要被文档化。
3. 内容要求（重意图轻语法）：
   - 注释至少覆盖：模块职责、核心类型语义、关键字段含义、关键方法用途。
   - 注释应解释“这段代码在表达什么业务语义或为什么这么写（Why）”，而不是机械复述代码语法（What）。
4. 特殊标记：若代码中有未完成事项或已知缺陷，必须使用 `// TODO:` 或 `// FIXME:` 明确标记，并简述原因。
5. 测试代码：测试代码可少写注释，优先通过具备描述性的测试函数名表达业务断言意图（如 `test_focus_panel_shows_file_tree_and_requests_focus`）。

### 3.5 命名规范

良好的命名是降低系统认知负荷的第一防线。所有的命名必须遵守“见名知意”的原则：

1. **拒绝无意义的泛指**：严禁使用 `data`、`info`、`item`（除非是 `ItemId` 这种有明确定义的领域词）、`process()`、`handle()` 这种没有任何实质信息的命名。名字必须精确描述其内容或行为。
2. **严守领域词汇表**：变量名必须与 `1.2 核心术语约定` 严格对齐。例如，代表标签页的变量必须叫 `tab` 而不能叫 `view` 或 `panel`；代表底层文本数据的必须叫 `buffer` 而不能叫 `file_data`。
3. **布尔值命名规范**：所有的 `bool` 类型变量和返回 `bool` 的函数，命名必须像一个完整的疑问句，通常以 `is_`、`has_`、`can_`、`should_` 开头（如 `is_active`、`has_selection`、`can_execute`）。
4. **动作方法前缀明确**：
   - `get_` / 无前缀：表示纯粹的读取，不应有副作用（Rust 中习惯省略 `get_` 直接用属性名，如 `.active_pane()`）。
   - `set_` / `update_`：表示修改状态。
   - `ensure_`：表示如果不存在则创建，如果存在则保持原样。
   - `toggle_`：表示布尔状态的反转。
5. **符合 Rust 习惯**：结构体、枚举、特征使用 `PascalCase`（大驼峰）；函数、变量、模块使用 `snake_case`（蛇形命名法）。

## 4. 命令与输入规范

1. `Command` 表达“用户意图”，与触发方式无关，是命令语义单一规范模型。
2. 命令 Key 与命令描述必须写入 `zom-protocol::command::kind` 的 `CommandKindSpec`；默认快捷键定义写入 `zom-input`，禁止在 `zom-gpui` 重复定义。
3. 快捷键、命令面板、菜单都应收敛到同一 `Command`，避免并行语义体系。
4. 输入映射（键位规则、上下文判定）统一在 `zom-input`，禁止在 `zom-gpui` 放输入解析实现。
5. `zom-runtime` 负责命令分发与副作用编排，避免多个入口分散执行逻辑。
6. 命令展示投影（如快捷键 hint）必须由 `zom-runtime::projection` 提供，`zom-gpui` 只消费投影，不直接读取 `zom-protocol` catalog 展示函数。
7. 高频编辑操作应优先提供直接快捷键或自然键入路径。
8. 低频功能应优先放入命令面板，而不是继续扩张快捷键数量。
9. 默认输入模型应保持无模式；若未来支持 Vim 行为，必须作为独立可选输入层接入，不能污染默认编辑模型。
10. 工具栏和界面提示应展示编辑上下文，如光标、选区、编码、换行格式，而不是模式名。
11. tooltip 文案和真实行为必须通过同一 `Command` 对齐，禁止“文案快捷键”与“真实快捷键”分离。

## 7. 日常开发检查清单

提交前自检（开发规范维度）：

1. 相关测试是否覆盖新增行为？
2. 是否存在用 `allow` 屏蔽 warning 的代码？

## 8. GPUI 视图与组件演进策略

在基于 GPUI 的 UI 架构中，将组件划分为 `View` 还是纯函数（`fn render`），核心取决于 **状态管理** 和 **性能边界**。总体遵循 **“默认纯函数，按需升 View”** 的渐进式策略。

### 8.1 必须升级为 `View` 的判定标准

1. **需要维护“局部交互状态”时（高内聚原则）**
   - 当组件需要自己维护不影响应用全局的交互数据（如面板宽度 `width`、拖拽状态 `is_dragging`、局部菜单的展开/收起状态）时，应升级为 `View`。
   - **反例**：将面板内部的拖拽状态或菜单开关强行“提升”到全局 `ZomRootView` 中，导致全局状态臃肿。
2. **需要阻断不必要的重绘时（性能隔离原则）**
   - `View` 是 GPUI 中最小的独立更新（`cx.notify()`）单元。高频触发 UI 更新的区域必须通过 `View` 进行隔离，防止局部更新导致整个应用根节点全量重绘。
   - **典型场景**：编辑器核心区（每次敲击键盘）、终端（高频日志流输出）。

### 8.2 分层组件的演进路线

1. **重型面板（必须是 View）**
   - **文件树 (FileTreePanel)**：包含拖拽调整宽度、层级展开/折叠、选中状态，具有强烈的局部交互属性。
   - **编辑器核心区 (EditorSurface)**：高频输入区，未来其内部的 Tab 管理、文本行渲染、多光标等都需要极端的性能优化和重绘隔离。
   - **终端面板 (TerminalPanel)**：需要独立接管高频的 PTY 渲染更新。
2. **边缘面板（按需演进：先纯函数 -> 后 View）**
   - **标题栏 (TitleBar) / 工具栏 (ToolBar)**：在初期仅作为“傻瓜式”的数据展示层时，保持纯函数 `fn render` 即可，代码最轻。
   - **演进契机**：当需要为其增加复杂的独立交互（如点击弹出独立作用域的设置菜单）时，再重构升级为 `View`。
3. **轻量无状态组件（永远保持纯函数）**
   - **基础 UI 元素 (Chip, Tooltip, 纯 Icon 渲染)**：永远保持为无状态的纯函数返回 `impl IntoElement`。
   - **严禁**：将这类基础原子组件封装为 `View`，否则会产生灾难性的性能开销和样板代码。

## 9. UI 布局规范：内容撑开高度 (Content-driven Height)

### 9.1 核心原则
在构建具备弹性和适应性的 UI 组件（特别是容器类组件如 Bar、Panel、List Item）时，**强烈建议使用内容撑开高度的设计模式，而避免硬编码固定高度像素（Magic Numbers）。**

### 9.2 为什么这样做？
1. **自动适配字号与缩放**：当用户调整全局字号（Font Size）或界面缩放比例时，基于内容撑开的容器会自动调整高度，包裹住内部内容，避免内容被截断或溢出。
2. **一致的视觉节奏**：通过对称的内边距（Padding）配合 Flex 布局，可以自然实现内部元素的垂直居中，且留白在不同组件间保持高度一致。
3. **极简代码**：消除了大量用于“拼凑”固定高度的计算逻辑和常量定义。

### 9.3 实践指南

#### A. 废弃硬编码高度
**禁止**在容器级别定义固定的 `height`。
```rust
// 反例：高度被定死，内部内容（如字号）变化时会产生布局挤压
div().h(px(28.0)).flex().items_center()
```

#### B. 拥抱 Padding 撑开
利用内容（文字、图标、胶囊）的自然高度，配合上下对称的内边距（Padding Y）来决定容器高度。
```rust
// 正式示范：使用 py (Padding Y) 撑开
div().w_full().py(px(SPACE_1)).flex().items_center()
```

#### C. 动态高度容器内的绝对定位 (电梯井模式)
如果容器高度是动态的，而内部子元素需要绝对定位且垂直居中，应采用“占满高度并 Flex 居中”的策略：
```rust
div()
    .absolute()
    .top_0()    // 贴顶
    .bottom_0() // 贴底，强制拉伸至父容器高度
    .flex()
    .items_center() // 在拉伸后的空间内垂直居中
    .child(...)
```

## 10. UI Token 规范：状态阶梯（灰度阶梯）

在 UI 层，除了颜色值本身，还存在大量“状态强弱”的连续值（如透明度、留白密度、控件热区大小）。  
为了避免魔法数散落，统一采用“**语义阶梯**”设计：**组件只表达语义，不直接写数值**。

### 10.1 核心原则

1. **语义优先，不暴露字面量**
   - 组件中禁止直接写 `0.0/1.0` 这类状态值（例如 `.opacity(0.0)`）。
   - 组件中应使用主题 token（如 `opacity::OPACITY_HIDDEN`）。
2. **维度分层，职责单一**
   - `color`：负责颜色语义（前景/背景/边框/状态色）。
   - `opacity`：负责可见性与状态强弱（隐藏/弱显/全显）。
   - `size`：负责几何尺寸（spacing/font/icon/control/layout）。
3. **禁止把组件名写进全局 token**
   - 不推荐 `PANE_XXX`、`FILE_TREE_XXX` 这类“组件绑定型”全局 token。
   - 推荐跨组件复用语义（如 `CONTROL_XS`、`SPACE_1`、`OPACITY_VISIBLE`）。
4. **平台常量允许例外，但必须标注来源**
   - 像 macOS 红绿灯直径这类平台规范值可保留常量。
   - 必须在注释中说明“平台约束”，避免被误当成可随意改动的样式值。

### 10.2 推荐命名方式

1. **颜色（`theme/color.rs`）**
   - 示例：`COLOR_BG_APP`、`COLOR_FG_MUTED`、`COLOR_BORDER`
2. **透明度（`theme/opacity.rs`）**
   - 示例：`OPACITY_HIDDEN`、`OPACITY_VISIBLE`
   - 后续可扩展：`OPACITY_SUBTLE`、`OPACITY_DISABLED`
3. **尺寸（`theme/size.rs`）**
   - 示例：`SPACE_1..N`、`ICON_SM/MD`、`CONTROL_XS`
   - 宏观布局：`PANEL_WIDTH_MIN/MAX`、`WINDOW_WIDTH/HEIGHT`

### 10.3 落地规则（代码评审必查）

1. 看到 `.opacity(0.0)`、`.opacity(1.0)`：必须改为 `theme/opacity` token。
2. 看到 `px(4.0)`、`px(8.0)` 等重复像素：优先抽到 `theme/size` 语义 token。
3. 看到组件内重复出现的状态值：优先上提到对应主题文件，而不是继续就地复制。
4. 对于一次性、仅出现一次且短期不复用的值：允许局部常量，但必须带注释解释理由。

### 10.4 示例

```rust
// 反例：魔法数散落在组件内部
.opacity(0.0)
.group_hover(group_id.to_string(), |style| style.opacity(1.0))

// 正例：使用状态阶梯 token
.opacity(opacity::OPACITY_HIDDEN)
.group_hover(group_id.to_string(), |style| style.opacity(opacity::OPACITY_VISIBLE))
```
