# Zom 通知系统设计与契约

版本：`v0.1`  
状态：`Active`（当前通知系统 SoT）  
适用范围：`zom-protocol`、`zom-input`、`zom-runtime`、`zom-gpui`

---

## 1. 目标

本契约用于统一 Zom 的通知系统设计，解决以下问题：

1. 减少高频弹窗对编辑流程的打断。
2. 保证关键通知可追溯、可筛选、可回看。
3. 把“通知策略”收敛到 runtime，避免 UI 层分散决策。

---

## 2. 产品方案（混合模式）

Zom 的通知系统统一采用三层通道：

1. `Toast`：瞬时提醒，面向“需要立即感知”的事件。
2. `NotificationPanel`：通知中心，面向“可追溯历史”与排查。
3. `StatusBar`：持续状态，面向“长生命周期进度”与后台任务。

该方案是“主流 IDE 分层模型”和“Zed 场景化提醒”之间的混合实现：  
默认少打扰，但保留完整历史与工程可观测性。

---

## 3. 跨 crate 协作边界

### 3.1 `zom-protocol`（语义层）

1. 定义通知相关命令语义与 `CommandKindSpec`。
2. 定义通知筛选、清理、已读等用户意图入口。
3. 不承载通知策略、去重算法、定时逻辑。

### 3.2 `zom-input`（输入层）

1. 维护通知命令默认快捷键与作用域。
2. 输出 `CommandInvocation`，不直接操作通知状态。

### 3.3 `zom-runtime`（编排层，主 crate）

1. 维护通知领域状态（列表、未读计数、活跃 toast、状态栏项目）。
2. 维护通知分发策略（事件 -> 通道）。
3. 维护去重、聚合、自动消失、已读回写等规则。
4. 提供通知展示 projection 给 `zom-gpui` 消费。

### 3.4 `zom-gpui`（渲染层）

1. 只渲染 toast / panel / status bar。
2. 只消费 runtime projection，不在组件内做通知级别决策。
3. 用户交互动作（清空、已读、重试）统一回到命令链路。

---

## 4. 领域模型契约（runtime）

### 4.1 基础类型

1. `NotificationSeverity`：`Info | Warning | Error`。
2. `NotificationChannel`：`Toast | Panel | StatusBar`。
3. `NotificationSource`：事件来源（如 `Workspace`、`FileTree`、`Lsp`、`Task`、`Debug`）。

### 4.2 事件与记录

1. `NotificationEvent`：一次领域事件输入，至少包含 `severity`、`source`、`message`。
2. `DesktopNotification`：持久化记录，建议包含：
   - `id`
   - `severity`
   - `source`
   - `message`
   - `created_at_ms`
   - `is_read`
   - `dedupe_key`
   - `occurrence_count`
3. `StatusItem`：持续状态项，建议包含 `key`、`label`、`progress`、`is_error`。

---

## 5. 分发策略契约（runtime）

默认策略如下：

1. `Error`：进入 `Toast + Panel`；若为持续故障，同时写 `StatusBar`。
2. `Warning`：进入 `Toast + Panel`。
3. `Info`：默认只进 `Panel`；仅“用户主动触发动作的结果反馈”可进 `Toast`。
4. `Progress` 类事件：优先进 `StatusBar`，完成或失败时写 `Panel` 摘要。

去重与聚合规则：

1. 同 `dedupe_key` 在短时间窗口内重复出现时，更新 `occurrence_count`，不重复弹 toast。
2. 聚合项文案必须可读，如 `5 tasks completed`，并可跳转详情。

---

## 6. 生命周期与不变量

### 6.1 生命周期

1. 事件产生：业务代码构造 `NotificationEvent`。
2. 策略分发：runtime 决定目标通道。
3. UI 展示：gpui 根据 projection 渲染。
4. 用户处理：已读、清空、跳转通过命令回流 runtime。

### 6.2 不变量

1. 不能只弹 toast 不写历史（需追溯的事件必须入 panel）。
2. 不能由 gpui 组件自行决定通知等级和分发通道。
3. 不能出现“面板已清空但未读计数未同步”的状态漂移。

---

## 7. 命令与输入契约

通知系统命令命名采用：
`workspace.notification.<verb>_<object>[.<target>]`

建议最小命令集：

1. `workspace.notification.focus_panel`
2. `workspace.notification.mark_all_read`
3. `workspace.notification.clear_all`
4. `workspace.notification.clear_read`
5. `workspace.notification.focus_unread_error`

说明：

1. 命令语义定义在 `zom-protocol::command::kind`。
2. 默认快捷键定义在 `zom-input`。
3. 执行分发统一走 `zom-runtime::state::DesktopAppState::handle_command`。

---

## 8. UI 表现契约（gpui）

### 8.1 Toast

1. 仅展示短文案与最多 1 个主操作。
2. 支持按等级差异化停留时长。
3. 不承载长文本日志。

### 8.2 NotificationPanel

1. 默认按时间倒序。
2. 支持 `All / Unread / Error` 筛选。
3. 支持 `mark all read` 与 `clear` 操作。

### 8.3 StatusBar

1. 展示持续状态与进度，不作为历史日志。
2. 点击状态项可触发对应命令（如打开通知面板或任务详情）。

---

## 9. 测试契约

1. `zom-runtime`：策略分发、去重聚合、已读计数、清空行为单测。
2. `zom-input`：通知命令默认键位解析测试。
3. `zom-gpui`：toast/panel/status bar 最小渲染与焦点回放烟雾测试。
4. 集成测试：从 `Keystroke -> Command -> NotificationState -> Projection` 的闭环验证。

---

## 10. 渐进落地计划

1. Phase 1：扩展 runtime 模型与策略，不改变现有交互路径。
2. Phase 2：补充通知命令、筛选和已读能力。
3. Phase 3：接入 status bar 持续状态通道并压降 info toast。
4. Phase 4：补齐测试与文档示例，冻结 v1 规则。

---

## 11. PR 验收清单

1. 是否把通知策略收敛在 `zom-runtime` 而非 `zom-gpui`？
2. 是否存在重复语义或 1:1 镜像类型？
3. 是否同步更新了 `CommandKindSpec`、默认快捷键、projection 与测试？
4. 是否避免引入新的“全局高频打断”行为？
