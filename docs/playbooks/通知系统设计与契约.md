# Zom 短时提示（Toast）系统设计与契约

版本：`v0.2`  
状态：`Active`（当前 Toast SoT）  
适用范围：`zom-protocol`、`zom-input`、`zom-runtime`、`zom-gpui`

---

## 1. 目标

本契约用于统一 Zom 的短时提示（Toast）设计，确保：

1. 反馈即时且低打断。
2. 规则集中在 runtime，避免 UI 分散决策。
3. 不引入通知中心/通知历史语义，保持实现轻量。

---

## 2. 产品方案（Toast-Only）

Zom 当前仅保留一个提示通道：`Toast`。

1. `Toast` 用于短时反馈，不承载历史回看。
2. 不提供通知面板、未读计数、筛选、已读、清空历史等能力。
3. 不提供通知状态栏常驻位作为通知系统组成部分。

---

## 3. 跨 crate 协作边界

### 3.1 `zom-protocol`（语义层）

1. 保持通用命令与焦点语义。
2. 不定义通知中心命令（如 `workspace.notification.*`）。
3. 不定义 toast 分发策略。

### 3.2 `zom-input`（输入层）

1. 维护通用命令快捷键。
2. 不维护通知中心相关默认键位。
3. 不直接操作 toast 状态。

### 3.3 `zom-runtime`（编排层，主 crate）

1. 持有 toast 领域状态：`active_toast`、`next_toast_id`。
2. 持有 toast 输入事件：`DesktopToastEvent`。
3. 维护唯一分发策略：事件是否应展示 toast。

### 3.4 `zom-gpui`（渲染层）

1. 只渲染当前 `active_toast`。
2. 只消费 runtime 状态，不判定分发等级策略。
3. 负责定时自动清除，并在清除前按 toast id 做二次核对。

---

## 4. 领域模型契约（runtime）

### 4.1 基础类型

1. `DesktopToastLevel`：`Info | Warning | Error`。
2. `DesktopToastEvent`：`level`、`message`、`is_user_initiated`。
3. `DesktopToast`：`id`、`level`、`message`。

### 4.2 显示策略

1. `Error`：总是显示 toast。
2. `Warning`：总是显示 toast。
3. `Info`：仅当 `is_user_initiated = true` 时显示 toast。

---

## 5. 生命周期与不变量

### 5.1 生命周期

1. 业务代码构造 `DesktopToastEvent`。
2. runtime 执行 `publish_toast_event` 并更新 `active_toast`。
3. gpui 渲染 toast 并安排自动清除。
4. 到期时若当前 `active_toast.id` 仍匹配，则清除。

### 5.2 不变量

1. 不保留通知中心专属能力：列表、筛选、已读、未读计数、历史清理。
2. 不保留来源分通道策略（如 `source -> panel/status bar`）。
3. 不在 gpui 组件内做“是否该显示 toast”的等级判断。
4. 不允许旧定时器清除新 toast（必须 id 比对）。

---

## 6. 命令与输入契约

1. Toast 不额外引入 `workspace.toast.*` 命令族。
2. Toast 由业务命令执行结果触发，不直接暴露“toast 管理命令”。
3. 若未来新增 toast 用户操作入口，需先更新本契约并补跨 crate 测试。

---

## 7. 测试契约

1. `zom-runtime`：覆盖 `Info` 用户触发门控、`Warning/Error` 直出、toast id 递增与清除。
2. `zom-gpui`：覆盖“写入 toast 后存在待自动清除标记”的最小链路。
3. 回归要求：禁止出现通知中心相关命令、状态与视图残留。

---

## 8. PR 验收清单

1. 是否新增了通知中心语义（如 panel/list/read/unread）？若是，应拒绝。
2. 是否把 toast 策略留在 runtime，而不是散落到 gpui？
3. 是否出现 `notification_*` 命名残留？
4. 是否同步更新测试与文档 SoT？
