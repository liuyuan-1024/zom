# Zom 开发规范手册

版本：`v0.1`  
适用范围：`/Users/liuyuan/project/zom` 全仓库  
目标：在持续演进为 IDE 的过程中，保持 **高内聚、低耦合、低跳转成本**。

---

## 1. 设计原则

1. 单一职责：每个 crate 只有一个主要变化原因。
2. 依赖单向：低层不依赖高层，禁止循环依赖。
3. 功能聚集：一个功能优先在一个主 crate 内完成，减少跨 crate 改动。
4. 协议稳定：`zom-core` 作为共享语言层，保持克制、稳定。
5. UI 隔离：GPUI 类型和实现不泄漏到内核层。
6. 拒绝过度隔离与 1:1 冗余：UI 隔离旨在禁止内核层了解渲染细节（如颜色、SVG路径），而非禁止 UI 层认识应用层的语义类型。严禁在 UI 层重新定义一套与底层 1:1 镜像的枚举并编写冗余转换代码。

### 1.1 产品与输入原则

1. Zom 的输入设计不采用 Vim 式模式切换作为核心交互模型。
2. 默认输入体验应接近 Zed：直接编辑、低心智负担、尽量减少“当前处于什么模式”的负担。
3. 高频操作必须支持全键盘直达，不要求用户频繁切换到鼠标。
4. 低频操作统一通过命令面板承载，避免为低频功能堆积复杂快捷键体系。
5. 工具栏、提示文案、命名和交互设计中，不应引入 `NORMAL`、`INSERT` 这类 Vim 模式语言，除非未来明确实现可选的 Vim 兼容层。
6. 命令系统表达的是“用户意图”，不是“模式状态”；输入系统表达的是“按键如何映射到意图”，不是“当前进入了哪种输入模式”。

### 1.2 核心术语约定

为了在代码命名、架构设计和日常沟通中保持一致，避免概念混淆，特约定以下核心界面术语（代码中的变量名和结构体名应严格遵守此对应关系）：

* **Workspace（工作台/工作区）**：最外层的运行环境。负责管理所有的窗口、面板和分割布局，是整个 IDE 的顶级容器。
* **Pane（窗格）**：带有标签页和具体内容展示区的独立块。一个 Workspace 可以被分割成多个 Pane（例如常见的左右分屏或上下分屏）。
* **Tab（标签页）**：Pane 顶部的导航条元素，用于在 Pane 内部的多个打开项之间进行切换。

## 2. 推荐 crate 划分（当前到目标）

### 2.1 当前基础结构

- `zom-core`
- `zom-text`
- `zom-input`
- `zom-editor`
- `zom-workspace`（预留）
- `zom-app`
- `zom-gpui`
- `apps/zom-desktop`

### 2.2 目标结构（推荐）

1. `zom-core`：基础类型、命令语义、稳定协议。
2. `zom-text`：文本存储和索引（rope/piece table、offset 映射）。
3. `zom-input`：输入映射（keymap、上下文解析、输入协议到命令语义）。
4. `zom-editor`：编辑行为（选择、光标、事务、undo/redo、搜索替换）。
5. `zom-app`：应用编排（command dispatch、状态协调、服务注入）。
6. `zom-gpui`：UI 适配与渲染（事件桥接、组件绘制）。
7. `apps/zom-desktop`：桌面入口（启动、依赖装配）。

### 2.3 可选后拆分

- `zom-workspace`：pane/tab/project 等工作台模型稳定后再独立。
- `zom-platform`：文件系统、剪贴板、watcher、进程等系统能力变复杂后再独立。

## 3. 依赖规则（强约束）

推荐依赖方向：

```text
zom-core
  ↑
zom-text
  ↑
zom-editor      zom-input
   ↑             ↑
   └─────── zom-app ───────┘
  ↑
zom-gpui
  ↑
apps/zom-desktop
```

补充规则：

1. `zom-core` 不能依赖任何业务 crate。
2. `zom-editor` 不能依赖 `zom-gpui`、`apps/*`。
3. `zom-gpui` 可以依赖 `zom-app`，但不能反向要求 `zom-app` 知道 GPUI 细节。
4. 所有跨 crate 通信，优先使用 `zom-core` 中定义的稳定类型与命令协议。
5. 出现循环依赖时，优先抽协议到低层 crate，而不是临时“互相引用”。

## 4. 各 crate 职责边界

### `zom-core`

- 放：`Position`、`Range`、`Selection`、`Id`、`Command`、方向/轴等。
- 不放：UI 类型、文件系统实现、复杂业务状态机。

### `zom-text`

- 放：文本结构、行列索引、增量变更应用。
- 不放：命令分发、窗口与面板状态。

### `zom-input`

- 放：按键映射、输入上下文解析、输入协议到命令/文本插入的分辨。
- 不放：GPUI 事件原始类型、编辑器文本算法实现。

### `zom-editor`

- 放：编辑动作和文本行为实现。
- 不放：GPUI 事件对象、pane/tab 布局管理。

### `zom-app`

- 放：用例编排、命令总线、跨模块协调。
- 不放：具体绘制代码。

### `zom-gpui`

- 放：视图、输入事件适配、渲染层状态绑定。
- 不放：底层文本算法。

### `apps/zom-desktop`

- 放：启动流程、依赖注入、运行时配置。
- 不放：核心编辑逻辑。

## 5. Crate 边界契约（必须遵守）

本节是边界判定的最终标准。评审冲突时，以本节为准。

### 5.1 边界总表

| Crate | 可以依赖 | 拥有（Owner） | 禁止拥有 | 对外暴露 |
|---|---|---|---|---|
| `zom-core` | 无业务 crate | 基础值对象、命令语义、强类型 ID | UI、文件系统实现、业务状态机 | 稳定类型与协议 |
| `zom-text` | `zom-core` | 文本存储、索引、位置映射 | 命令分发、面板/窗口状态 | 文本读写与映射 API |
| `zom-input` | `zom-core` | 按键映射、输入上下文解析 | 编辑算法、UI 原始事件对象 | 输入解析 API |
| `zom-editor` | `zom-core`、`zom-text` | 编辑事务、光标/选区、undo/redo、搜索替换 | GPUI 类型、窗口编排 | 编辑行为 API |
| `zom-app` | `zom-core`、`zom-text`、`zom-input`、`zom-editor`（后续可接 `zom-workspace`/`zom-platform`） | 命令分发、用例编排、跨模块协调 | 具体渲染实现 | 应用服务入口 |
| `zom-gpui` | `zom-app`、`zom-core` | 视图组件、事件适配、UI 状态绑定 | 文本底层算法、编辑内核实现 | UI 入口与组件 |
| `apps/zom-desktop` | `zom-gpui`、`zom-app` | 启动、注入、运行时配置 | 业务规则、文本算法 | 可执行入口 |

### 5.2 放置决策树

1. 是基础类型/协议并且跨层复用？放 `zom-core`。
2. 是文本结构、索引、位置映射？放 `zom-text`。
3. 是按键映射、输入上下文判定、输入到命令分辨？放 `zom-input`。
4. 是编辑行为（移动、删除、撤销重做、多光标）？放 `zom-editor`。
5. 是“调用谁做什么”的编排逻辑？放 `zom-app`。
6. 是输入事件、组件绘制、UI 生命周期？放 `zom-gpui`。
7. 是程序启动与依赖装配？放 `apps/zom-desktop`。

### 5.3 反例（禁止）

1. 在 `zom-editor` 里直接使用 `gpui::*` 类型。
2. 在 `zom-gpui` 里实现 rope、行列映射、undo 算法。
3. 在 `zom-core` 放快捷键解析、命令分发状态机（应放 `zom-input`/`zom-app`）。
4. 在 `apps/zom-desktop` 写编辑行为逻辑。
5. 任何“为快一点先互相引用”导致的循环依赖。
6. 禁止镜像枚举：禁止在 zom-gpui 中创建与 zom-app 完全对应的图标枚举（如 UiIcon 转 AppIcon）。UI 层应直接 match 应用层枚举并返回 UI 资产。

### 5.4 边界例外机制

必须临时跨边界时，需同时满足：

1. 建立一条 ADR 记录（`docs/adr/`）。
2. 标注“临时”与回收时间点。
3. 在两个迭代内回收到标准边界，否则禁止继续叠加。

## 6. 降低“跨 crate 跳转”的开发策略

1. 每个功能设一个“主 crate”：
   - 文本编辑行为默认主 crate 是 `zom-editor`。
   - 文本底层性能优化默认主 crate 是 `zom-text`。
   - 视图交互默认主 crate 是 `zom-gpui`。
2. 非必要不改 `zom-core`。`core` 变更需要明确收益和影响面。
3. 一次需求若连续 3 次都需改 3 个以上 crate，必须复盘边界是否拆错。
4. `zom-editor` 内按 feature 分目录，避免按技术层碎片化文件。
5. PR 必须声明：
   - 主 crate（唯一）
   - 次要改动 crate（尽量 <= 2）
   - 是否触及 `zom-core`

## 7. 代码组织规范

### 7.1 模块组织

- 优先按“领域功能”组织，而非按“工具类别”组织。
- 示例（`zom-editor`）：
  - `features/cursor/*`
  - `features/selection/*`
  - `features/history/*`
  - `features/search/*`

### 7.2 API 设计

1. crate 对外暴露统一入口（facade），避免调用方了解内部结构。
2. 参数优先传领域类型，不传裸 `u64/String`（使用强类型 ID）。
3. 明确同步/异步边界，避免隐式阻塞 UI。

### 7.3 错误处理

1. 公共层返回结构化错误，避免字符串拼接错误。
2. 禁止在库代码中滥用 `unwrap/expect`。
3. 禁止使用 `#[allow(...)]`、`#![allow(...)]` 或其他形式主动屏蔽 warning。
4. 出现 warning 时，应修复根因，而不是通过 `allow` 压过去。

### 7.4 注释规范

1. 所有生产代码都必须补充注释。
2. 注释至少覆盖：模块职责、核心类型语义、关键字段含义、关键方法用途。
3. 注释应解释“这段代码在表达什么语义”，而不是机械复述语法。
4. 测试代码可以少写注释，优先通过测试名表达意图。

### 7.5 命名规范

良好的命名是降低系统认知负荷的第一防线。所有的命名必须遵守“见名知意”的原则：

1. **拒绝无意义的泛指**：严禁使用 `data`、`info`、`item`（除非是 `ItemId` 这种有明确定义的领域词）、`process()`、`handle()` 这种没有任何实质信息的命名。名字必须精确描述其内容或行为。
2. **严守领域词汇表**：变量名必须与 `1.2 核心术语约定` 严格对齐。例如，代表标签页的变量必须叫 `tab` 而不能叫 `view` 或 `panel`；代表底层文本数据的必须叫 `buffer` 而不能叫 `file_data`。
3. **布尔值命名规范**：所有的 `bool` 类型变量和返回 `bool` 的函数，命名必须像一个完整的疑问句，通常以 `is_`、`has_`、`can_`、`should_` 开头（如 `is_active`、`has_selection`、`can_execute`）。
4. **动作方法前缀明确**：
   - `get_` / 无前缀：表示纯粹的读取，不应有副作用（Rust 中习惯省略 `get_` 直接用属性名，如 `.active_pane()`）。
   - `set_` / `update_`：表示修改状态。
   - `ensure_`：表示如果不存在则创建，如果存在则保持原样。
   - `toggle_`：表示布尔状态的反转。
5. **符合 Rust 习惯**：结构体、枚举、特征使用 `PascalCase`（大驼峰）；函数、变量、模块使用 `snake_case`（蛇形命名法）。

## 8. 命令与输入规范

1. `Command` 表达“用户意图”，与触发方式无关。
2. 快捷键、命令面板、菜单都应收敛到同一命令协议。
3. 输入映射建议独立成 `zom-input`（或至少独立模块），避免污染 `core`。
4. 命令分发统一由 `zom-app` 协调，避免多个入口分散执行逻辑。
5. 高频编辑操作应优先提供直接快捷键或自然键入路径。
6. 低频功能应优先放入命令面板，而不是继续扩张快捷键数量。
7. 默认输入模型应保持无模式；若未来支持 Vim 行为，必须作为独立可选输入层接入，不能污染默认编辑模型。
8. 工具栏和界面提示应展示编辑上下文，如光标、选区、编码、换行格式，而不是模式名。

## 9. 测试规范

1. `zom-core`：纯单元测试，覆盖基础类型行为。
2. `zom-text`：边界与性能相关测试（大文件、行列映射、增量修改）。
3. `zom-editor`：行为测试（编辑事务、undo/redo、多光标）。
4. `zom-app`：集成测试（命令分发和跨模块协同）。
5. `zom-gpui`：以最小烟雾测试为主，核心逻辑不在 UI 层验证。

建议执行：

```bash
./scripts/check-boundaries.sh
cargo test --workspace
```

## 10. 迭代与拆分策略

当前阶段（编辑器核心）采用“先聚后拆”：

1. 先让编辑闭环稳定（打开/编辑/撤销重做/保存/搜索替换）。
2. 观测高频改动点，再拆分 `zom-text`、`zom-platform`、`zom-workspace`。
3. 拆分标准：
   - 职责独立且接口稳定
   - 能显著降低主 crate 复杂度
   - 拆后能减少而非增加日常跳转

## 12. 日常开发检查清单

提交前自检：

1. 这次改动的主 crate 是否明确？
2. 是否引入了不必要的跨 crate 变更？
3. 是否破坏了依赖单向原则？
4. 是否通过边界守卫脚本（`./scripts/check-boundaries.sh`）？
5. 是否改动了 `zom-core`，且有充分理由？
6. 相关测试是否覆盖新增行为？
7. 是否把 UI 细节泄漏到了内核层？
8. 是否存在用 `allow` 屏蔽 warning 的代码？
9. 新增或修改的生产代码是否已补充注释？
10. 本次交互设计是否违背“无模式 + 高频键盘直达 + 低频命令面板”的产品原则？
11. 是否为了所谓的“解耦”，在顶层 crate 定义了与底层完全 1:1 的冗余枚举或结构体，并写了无意义的转换代码？

## 14. GPUI 视图与组件演进策略

在基于 GPUI 的 UI 架构中，将组件划分为 `View` 还是纯函数（`fn render`），核心取决于 **状态管理** 和 **性能边界**。总体遵循 **“默认纯函数，按需升 View”** 的渐进式策略。

### 14.1 必须升级为 `View` 的判定标准

1. **需要维护“局部交互状态”时（高内聚原则）**
   - 当组件需要自己维护不影响应用全局的交互数据（如面板宽度 `width`、拖拽状态 `is_dragging`、局部菜单的展开/收起状态）时，应升级为 `View`。
   - **反例**：将面板内部的拖拽状态或菜单开关强行“提升”到全局 `ZomRootView` 中，导致全局状态臃肿。
2. **需要阻断不必要的重绘时（性能隔离原则）**
   - `View` 是 GPUI 中最小的独立更新（`cx.notify()`）单元。高频触发 UI 更新的区域必须通过 `View` 进行隔离，防止局部更新导致整个应用根节点全量重绘。
   - **典型场景**：编辑器核心区（每次敲击键盘）、终端（高频日志流输出）。

### 14.2 分层组件的演进路线

1. **重型面板（必须是 View）**
   - **文件树 (FileTreePanel)**：包含拖拽调整宽度、层级展开/折叠、选中状态，具有强烈的局部交互属性。
   - **编辑器核心区 (EditorSurface)**：高频输入区，未来其内部的 Tab 管理、文本行渲染、多光标等都需要极端的性能优化和重绘隔离。
   - **终端面板 (TerminalPanel)**：需要独立接管高频的 PTY 渲染更新。
2. **边缘面板（按需演进：先纯函数 -> 后 View）**
   - **标题栏 (TitleBar) / 工具栏 (ToolBar)**：在初期仅作为“傻瓜式”的数据展示层时，保持纯函数 `fn render` 即可，代码最轻。
   - **演进契机**：当需要为其增加复杂的独立交互（如点击弹出独立作用域的设置菜单）时，再重构升级为 `View`。
3. **轻量无状态组件（永远保持纯函数）**
   - **基础 UI 元素 (Chip, Tooltip, 纯 Icon 渲染)**：永远保持为无状态的纯函数返回 `impl IntoElement`。
   - **严禁**：将这类基础原子组件封装为 `View`，否则会产生灾难性的性能开销和样板代码。

## 15. UI 布局规范：内容撑开高度 (Content-driven Height)

### 15.1 核心原则
在构建具备弹性和适应性的 UI 组件（特别是容器类组件如 Bar、Panel、List Item）时，**强烈建议使用内容撑开高度的设计模式，而避免硬编码固定高度像素（Magic Numbers）。**

### 15.2 为什么这样做？
1. **自动适配字号与缩放**：当用户调整全局字号（Font Size）或界面缩放比例时，基于内容撑开的容器会自动调整高度，包裹住内部内容，避免内容被截断或溢出。
2. **一致的视觉节奏**：通过对称的内边距（Padding）配合 Flex 布局，可以自然实现内部元素的垂直居中，且留白在不同组件间保持高度一致。
3. **极简代码**：消除了大量用于“拼凑”固定高度的计算逻辑和常量定义。

### 15.3 实践指南

#### A. 废弃硬编码高度
**禁止**在容器级别定义固定的 `height`。
```rust
// 反例：高度被定死，内部内容（如字号）变化时会产生布局挤压
div().h(px(28.0)).flex().items_center()
```

#### B. 拥抱 Padding 撑开
利用内容（文字、图标、胶囊）的自然高度，配合上下对称的内边距（Padding Y）来决定容器高度。
```rust
// 正式示范：使用 py (Padding Y) 撑开
div().w_full().py(px(SPACE_1)).flex().items_center()
```

#### C. 动态高度容器内的绝对定位 (电梯井模式)
如果容器高度是动态的，而内部子元素需要绝对定位且垂直居中，应采用“占满高度并 Flex 居中”的策略：
```rust
div()
    .absolute()
    .top_0()    // 贴顶
    .bottom_0() // 贴底，强制拉伸至父容器高度
    .flex()
    .items_center() // 在拉伸后的空间内垂直居中
    .child(...)
```
