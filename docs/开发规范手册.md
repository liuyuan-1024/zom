# Zom 开发规范手册

版本：`v0.1`  
适用范围：`/Users/liuyuan/project/zom` 全仓库  
目标：在持续演进为 IDE 的过程中，保持 **高内聚、低耦合、低跳转成本**。

---

## 1. 设计原则

1. 单一职责：每个 crate 只有一个主要变化原因。
2. 依赖单向：低层不依赖高层，禁止循环依赖。
3. 功能聚集：一个功能优先在一个主 crate 内完成，减少跨 crate 改动。
4. 协议稳定：`zom-core` 作为共享语言层，保持克制、稳定。
5. UI 隔离：GPUI 类型和实现不泄漏到内核层。
6. 拒绝过度隔离与 1:1 冗余： UI 隔离的目的是“禁止内核/应用层知道 UI 的渲染细节（如颜色、字体、SVG路径）”，而不是“禁止 UI 层认识应用层的语义类型”。只要依赖方向合法（如 zom-gpui 依赖 zom-app），UI 层应直接使用下层暴露的语义枚举，禁止在 UI 层重新定义一套 1:1 的镜像类型并编写无意义的 From 转换代码。

### 1.1 产品与输入原则

1. Zom 的输入设计不采用 Vim 式模式切换作为核心交互模型。
2. 默认输入体验应接近 Zed：直接编辑、低心智负担、尽量减少“当前处于什么模式”的负担。
3. 高频操作必须支持全键盘直达，不要求用户频繁切换到鼠标。
4. 低频操作统一通过命令面板承载，避免为低频功能堆积复杂快捷键体系。
5. 工具栏、提示文案、命名和交互设计中，不应引入 `NORMAL`、`INSERT` 这类 Vim 模式语言，除非未来明确实现可选的 Vim 兼容层。
6. 命令系统表达的是“用户意图”，不是“模式状态”；输入系统表达的是“按键如何映射到意图”，不是“当前进入了哪种输入模式”。

## 2. 推荐 crate 划分（当前到目标）

### 2.1 当前基础结构

- `zom-core`
- `zom-text`
- `zom-input`
- `zom-editor`
- `zom-workspace`（预留）
- `zom-app`
- `zom-gpui`
- `apps/zom-desktop`

### 2.2 目标结构（推荐）

1. `zom-core`：基础类型、命令语义、稳定协议。
2. `zom-text`：文本存储和索引（rope/piece table、offset 映射）。
3. `zom-input`：输入映射（keymap、上下文解析、输入协议到命令语义）。
4. `zom-editor`：编辑行为（选择、光标、事务、undo/redo、搜索替换）。
5. `zom-app`：应用编排（command dispatch、状态协调、服务注入）。
6. `zom-gpui`：UI 适配与渲染（事件桥接、组件绘制）。
7. `apps/zom-desktop`：桌面入口（启动、依赖装配）。

### 2.3 可选后拆分

- `zom-workspace`：pane/tab/project 等工作台模型稳定后再独立。
- `zom-platform`：文件系统、剪贴板、watcher、进程等系统能力变复杂后再独立。

## 3. 依赖规则（强约束）

推荐依赖方向：

```text
zom-core
  ↑
zom-text
  ↑
zom-editor      zom-input
   ↑             ↑
   └─────── zom-app ───────┘
  ↑
zom-gpui
  ↑
apps/zom-desktop
```

补充规则：

1. `zom-core` 不能依赖任何业务 crate。
2. `zom-editor` 不能依赖 `zom-gpui`、`apps/*`。
3. `zom-gpui` 可以依赖 `zom-app`，但不能反向要求 `zom-app` 知道 GPUI 细节。
4. 所有跨 crate 通信，优先使用 `zom-core` 中定义的稳定类型与命令协议。
5. 出现循环依赖时，优先抽协议到低层 crate，而不是临时“互相引用”。

## 4. 各 crate 职责边界

### `zom-core`

- 放：`Position`、`Range`、`Selection`、`Id`、`Command`、方向/轴等。
- 不放：UI 类型、文件系统实现、复杂业务状态机。

### `zom-text`

- 放：文本结构、行列索引、增量变更应用。
- 不放：命令分发、窗口与面板状态。

### `zom-input`

- 放：按键映射、输入上下文解析、输入协议到命令/文本插入的分辨。
- 不放：GPUI 事件原始类型、编辑器文本算法实现。

### `zom-editor`

- 放：编辑动作和文本行为实现。
- 不放：GPUI 事件对象、pane/tab 布局管理。

### `zom-app`

- 放：用例编排、命令总线、跨模块协调。
- 不放：具体绘制代码。

### `zom-gpui`

- 放：视图、输入事件适配、渲染层状态绑定。
- 不放：底层文本算法。

### `apps/zom-desktop`

- 放：启动流程、依赖注入、运行时配置。
- 不放：核心编辑逻辑。

## 5. Crate 边界契约（必须遵守）

本节是边界判定的最终标准。评审冲突时，以本节为准。

### 5.1 边界总表

| Crate | 可以依赖 | 拥有（Owner） | 禁止拥有 | 对外暴露 |
|---|---|---|---|---|
| `zom-core` | 无业务 crate | 基础值对象、命令语义、强类型 ID | UI、文件系统实现、业务状态机 | 稳定类型与协议 |
| `zom-text` | `zom-core` | 文本存储、索引、位置映射 | 命令分发、面板/窗口状态 | 文本读写与映射 API |
| `zom-input` | `zom-core` | 按键映射、输入上下文解析 | 编辑算法、UI 原始事件对象 | 输入解析 API |
| `zom-editor` | `zom-core`、`zom-text` | 编辑事务、光标/选区、undo/redo、搜索替换 | GPUI 类型、窗口编排 | 编辑行为 API |
| `zom-app` | `zom-core`、`zom-text`、`zom-input`、`zom-editor`（后续可接 `zom-workspace`/`zom-platform`） | 命令分发、用例编排、跨模块协调 | 具体渲染实现 | 应用服务入口 |
| `zom-gpui` | `zom-app`、`zom-core` | 视图组件、事件适配、UI 状态绑定 | 文本底层算法、编辑内核实现 | UI 入口与组件 |
| `apps/zom-desktop` | `zom-gpui`、`zom-app` | 启动、注入、运行时配置 | 业务规则、文本算法 | 可执行入口 |

### 5.2 放置决策树

1. 是基础类型/协议并且跨层复用？放 `zom-core`。
2. 是文本结构、索引、位置映射？放 `zom-text`。
3. 是按键映射、输入上下文判定、输入到命令分辨？放 `zom-input`。
4. 是编辑行为（移动、删除、撤销重做、多光标）？放 `zom-editor`。
5. 是“调用谁做什么”的编排逻辑？放 `zom-app`。
6. 是输入事件、组件绘制、UI 生命周期？放 `zom-gpui`。
7. 是程序启动与依赖装配？放 `apps/zom-desktop`。

### 5.3 反例（禁止）

1. 在 `zom-editor` 里直接使用 `gpui::*` 类型。
2. 在 `zom-gpui` 里实现 rope、行列映射、undo 算法。
3. 在 `zom-core` 放快捷键解析、命令分发状态机（应放 `zom-input`/`zom-app`）。
4. 在 `apps/zom-desktop` 写编辑行为逻辑。
5. 任何“为快一点先互相引用”导致的循环依赖。
6. 在 zom-gpui 中创建与 zom-app / zom-core 语义 1:1 对应的镜像枚举。 错误做法： zom-app 定义了 ToolBarIcon::Search，zom-gpui 又定义一个完全一样的 ToolIcon::Search，然后写 impl From<ToolBarIcon> for ToolIcon。

### 5.4 边界例外机制

必须临时跨边界时，需同时满足：

1. 建立一条 ADR 记录（`docs/adr/`）。
2. 标注“临时”与回收时间点。
3. 在两个迭代内回收到标准边界，否则禁止继续叠加。

## 6. 降低“跨 crate 跳转”的开发策略

1. 每个功能设一个“主 crate”：
   - 文本编辑行为默认主 crate 是 `zom-editor`。
   - 文本底层性能优化默认主 crate 是 `zom-text`。
   - 视图交互默认主 crate 是 `zom-gpui`。
2. 非必要不改 `zom-core`。`core` 变更需要明确收益和影响面。
3. 一次需求若连续 3 次都需改 3 个以上 crate，必须复盘边界是否拆错。
4. `zom-editor` 内按 feature 分目录，避免按技术层碎片化文件。
5. PR 必须声明：
   - 主 crate（唯一）
   - 次要改动 crate（尽量 <= 2）
   - 是否触及 `zom-core`

## 7. 代码组织规范

### 7.1 模块组织

- 优先按“领域功能”组织，而非按“工具类别”组织。
- 示例（`zom-editor`）：
  - `features/cursor/*`
  - `features/selection/*`
  - `features/history/*`
  - `features/search/*`

### 7.2 API 设计

1. crate 对外暴露统一入口（facade），避免调用方了解内部结构。
2. 参数优先传领域类型，不传裸 `u64/String`（使用强类型 ID）。
3. 明确同步/异步边界，避免隐式阻塞 UI。

### 7.3 错误处理

1. 公共层返回结构化错误，避免字符串拼接错误。
2. 禁止在库代码中滥用 `unwrap/expect`。
3. 禁止使用 `#[allow(...)]`、`#![allow(...)]` 或其他形式主动屏蔽 warning。
4. 出现 warning 时，应修复根因，而不是通过 `allow` 压过去。

### 7.4 注释规范

1. 所有生产代码都必须补充注释。
2. 注释至少覆盖：模块职责、核心类型语义、关键字段含义、关键方法用途。
3. 注释应解释“这段代码在表达什么语义”，而不是机械复述语法。
4. 测试代码可以少写注释，优先通过测试名表达意图。

---

## 8. 命令与输入规范

1. `Command` 表达“用户意图”，与触发方式无关。
2. 快捷键、命令面板、菜单都应收敛到同一命令协议。
3. 输入映射建议独立成 `zom-input`（或至少独立模块），避免污染 `core`。
4. 命令分发统一由 `zom-app` 协调，避免多个入口分散执行逻辑。
5. 高频编辑操作应优先提供直接快捷键或自然键入路径。
6. 低频功能应优先放入命令面板，而不是继续扩张快捷键数量。
7. 默认输入模型应保持无模式；若未来支持 Vim 行为，必须作为独立可选输入层接入，不能污染默认编辑模型。
8. 工具栏和界面提示应展示编辑上下文，如光标、选区、编码、换行格式，而不是模式名。

## 9. 测试规范

1. `zom-core`：纯单元测试，覆盖基础类型行为。
2. `zom-text`：边界与性能相关测试（大文件、行列映射、增量修改）。
3. `zom-editor`：行为测试（编辑事务、undo/redo、多光标）。
4. `zom-app`：集成测试（命令分发和跨模块协同）。
5. `zom-gpui`：以最小烟雾测试为主，核心逻辑不在 UI 层验证。

建议执行：

```bash
./scripts/check-boundaries.sh
cargo test --workspace
```

## 10. 迭代与拆分策略

当前阶段（编辑器核心）采用“先聚后拆”：

1. 先让编辑闭环稳定（打开/编辑/撤销重做/保存/搜索替换）。
2. 观测高频改动点，再拆分 `zom-text`、`zom-platform`、`zom-workspace`。
3. 拆分标准：
   - 职责独立且接口稳定
   - 能显著降低主 crate 复杂度
   - 拆后能减少而非增加日常跳转

## 11. 架构变更流程（轻量 ADR）

涉及 crate 边界或依赖方向变化时，需新增一条简短记录到 `docs/adr/`：

1. 背景与问题
2. 方案对比
3. 决策与影响
4. 回滚条件

目标：保证后续回看时，能理解“为什么这么拆”。

---

## 12. 日常开发检查清单

提交前自检：

1. 这次改动的主 crate 是否明确？
2. 是否引入了不必要的跨 crate 变更？
3. 是否破坏了依赖单向原则？
4. 是否通过边界守卫脚本（`./scripts/check-boundaries.sh`）？
5. 是否改动了 `zom-core`，且有充分理由？
6. 相关测试是否覆盖新增行为？
7. 是否把 UI 细节泄漏到了内核层？
8. 是否存在用 `allow` 屏蔽 warning 的代码？
9. 新增或修改的生产代码是否已补充注释？
10. 本次交互设计是否违背“无模式 + 高频键盘直达 + 低频命令面板”的产品原则？
11. 是否为了所谓的“解耦”，在顶层 crate 定义了与底层完全 1:1 的冗余枚举或结构体，并写了无意义的转换代码？

---

## 13. 当前执行约定（立即生效）

1. 编辑器核心开发以 `zom-editor` 为主阵地。
2. `zom-core` 保持克制，新增类型优先评估复用价值。
3. 新功能默认不新增 crate，除非已满足拆分标准。
4. 后续若新增 `zom-text`，先迁移文本存储与索引，不迁移编辑行为。
